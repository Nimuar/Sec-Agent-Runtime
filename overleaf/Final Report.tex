\begin{abstract}

\begin{itemize}
    \item \textbf{Context:} Modern software architectures are transitioning from human-in-the-middle processes to fully autonomous machine-in-the-middle workflows. This shift is driven by breakthroughs in Agentic Artificial Intelligence, granting Large Language Models (LLMs) the ability to invoke APIs and manipulate file systems. However, allowing non-deterministic models direct control over execution environments introduces critical security vulnerabilities.
    \item \textbf{Objective:} Prior work has characterized the vulnerabilities of prompt injection and hallucination, but as yet we lack standardized architectures for deterministically constraining autonomous agents at the host environment level.
    \item \textbf{Method}: We identified key architectural vulnerabilities in agent deployment. To address this gap, we engineered a stateless, deterministic mediation layer based on a Positive Security Model. Logically decoupling an untrusted Python agent SDK from a trusted Node.js execution runtime via a local HTTP interface, we statically enforce security constraints by validating JSON-formatted agent proposals against strict Zod schemas.
    \item \textbf{Results}: We implemented an end-to-end prototype of the Security-Constrained Agent Runtime (SCR) and evaluated it against an aggregated dataset of 364 simulated interactions, including adversarial jailbreak payloads. The system yielded a 0.0 False Negative Rate, successfully blocking 100\% of malicious execution attempts (e.g., path traversal and arbitrary file writes). Stress testing with a larger test pool for our V3 system (7 to 117 tests) improved our robustness and revealed a highly restrictive posture with a 94.3\% False Positive Rate driven by rigid schema enforcement on malformed JSON, alongside an overall infrastructure System Fault Rate of 9.4\%. 
    \item \textbf{Conclusion}: The results demonstrate that a deterministic mediation layer can effectively neutralize adversarial intent and enforce strict security policies for autonomous agents. While the Positive Security Model currently restricts operational utility due to high sensitivity to formatting errors, separating agent intent from host execution provides a secure, deterministic, and auditable foundation for safely integrating LLMs into real-world software systems. 
\end{itemize}
\end{abstract}

\section{Introduction}

Modern software architectures are transitioning from human-in-the-middle processes to fully autonomous machine-in-the-middle workflows , driven by breakthroughs in Agentic Artificial Intelligence and Large Language Models (LLMs). As noted by Jin et al. \cite{jin2025llmsllmbasedagentssoftware} and Dong et al. \cite{dong2025surveycodegenerationllmbased}, LLMs are no longer limited to passive text generation; agents are increasingly integrated into production environments where they take intrusive actions such as invoking APIs, manipulating file systems, and managing system resources. While this promises to revolutionize software engineering, it introduces critical security risks by granting non-deterministic models direct control over execution environments.

Current research addresses this gap by framing the agent not as a tool, but as a potential threat vector. Wen et al. \cite{wen2024adaptivedeploymentuntrustedllms} propose a framework that explicitly treats the LLM Agent as an untrusted component, demonstrating that safety cannot be assumed but must be enforced externally. Their work shows that "wrapping" untrusted models in a rigorous protocol allows for the restriction of agentic actions based on risk, significantly reducing the success rate of distributed threats. Despite this theoretical foundation, most practical deployments still rely on probabilistic guardrails rather than deterministic constraints. As a result, a key open problem remains: how to enforce strict, pre-execution constraints on agent-generated actions within a production-grade runtime.

To address this gap, this project implements a Security-Constrained Agent Runtime designed to deterministically mediate all interactions between an untrusted agent and the execution environment. The project adopts a positive security model called Allowlisting, which rejects all actions by default unless they adhere to a predefined JSON schemas and policies. The investigation focuses on defining a Proposal Schema to validate agent intent, enforcing file-system sandboxing, limiting operations to specific file types, and implementing methods to track and maintaining an auditable execution lifecycle. 

To investigate and validate this deterministic mediation strategy, this project implements a stateless, local HTTP-based execution loop that isolates the non-deterministic LLM logic from the host system. The approach centers on treating the LLM as a completely untrusted Python client that must negotiate with a Node.js/TypeScript server. Rather than relying on dynamic, probabilistic guardrails, the project approaches the problem through static policy enforcement. By utilizing Zod to enforce strict JSON schemas on the ingress layer, the runtime structurally guarantees that all proposals are constrained to a designated set of allowed actions and directories before the underlying Node.js file system wrappers are ever invoked. This architecture ensures that every tool invocation is evaluated strictly on its semantic merit and recorded to an immutable audit log for post-hoc analysis.

The resulting prototype demonstrates that deterministic mediation can effectively constrain agent behavior in practice. Our implementation processes agent proposals through a structured validation and execution lifecycle that includes schema validation, action authorization, and controlled execution. Experimental testing shows that malformed requests, unauthorized actions, and sandbox violations are consistently rejected, while valid proposals successfully progress through the runtime lifecycle. Furthermore, repeated executions with identical inputs produce deterministic outputs, confirming the system’s ability to enforce consistent policy decisions while maintaining auditable and reproducible execution behavior.

\subsubsection{Adaptive Deployment of Untrusted LLMs Reduces Distributed Threats:}

\textbf{Prospective Goals:} This system addresses a fundamental problem with untrusted LLMs that often fly under the radar. Rather than targeting models that attempt to bypass safety features with a single catastrophic attack, this framework addresses the "distributed threat setting," where no individual action provides overwhelming evidence of misalignment. Instead, it is the amalgamation of these actions that leads to failure \cite{wen2024adaptivedeploymentuntrustedllms}.

The framework operates by using an adaptive macro-protocol that chooses among several different micro-protocols to apply, rather than just two. These micro-protocols are task-specific and use a simpler, more thoroughly tested trusted model to monitor and interact with the more capable but untrusted model. Ultimately, the macro-protocol retains the discretion to select which specific micro-protocol to implement for each task based on an evolving estimate of the untrusted model’s alignment over time. This system was also validated using the open-sourced APPS Backdoor Testbed for testing an validation.


\textbf{Technology:}
The \textbf{macro-model} operates as a high-level control mechanism that governs how untrusted LLM outputs are handled over time. Rather than directly executing model-generated actions, the macro-model maintains global oversight by selecting among multiple micro-model protocols based on an evolving assessment of the untrusted model’s behavior. This enables adaptive responses to gradual misalignment, particularly in distributed threat settings where no single action is sufficient to trigger intervention~\cite{wen2024adaptivedeploymentuntrustedllms}.


The \textbf{micro-model} increases granularity by acting on a single input at a time. It uses a trusted model in conjunction with an untrusted model to produce it's output. There are multiple micro-models that the macro-model uses and each generates a different output (y). 
Breaking the agent behavior into these smaller units makes individual interactions easier to inspect and test than analyzing a fully unconstrained agent as a whole. Previous work on agentic tool orchestration has shown that structuring agent behavior into well-defined protocols can make safety evaluation more manageable~\cite{chen2025agentguardsafetytoolorchestration, yao2023react}. However, this approach also adds design overhead, since each micro-model must be carefully defined and maintained to provide sufficient coverage.

\textbf{Domain: }
The research in the first paper focuses on defending against untrusted LLMs within a subdomain different from our project’s scope. It addresses the "distributed threat setting," where the Threat Model targets actions that might not be individually catastrophic—such as subtle backdoors—but could lead to massive failure as actions accumulate. \cite{wen2024adaptivedeploymentuntrustedllms}. This framework specifically employs an adaptive\textbf{ }macro-protocol model that updates its micro-model to find the most efficient course of action for long-term safety.In contrast, our project focuses on a deterministic\textbf{ }method of sanitization~\cite{watson2010capsicum}. We utilize explicit\textbf{ }whitelists, argument schemas, and tool-invocation policies to ensure that our system approves every type of action before it is allowed to pass. 

Our \textbf{Threat Model} assumes an adversarial environment in which malicious inputs 
are submitted by an untrusted LLM agent in an attempt to bypass our policy 
enforcement checker. This model is divided into two attack classes. The first, 
\textbf{Adversarial Payloads}, encompasses inputs that attempt to circumvent 
policy checks and exploit boundary conditions. The second, \textbf{Malformed 
Inputs}, covers inputs that violate policy schema enforcement. Our pipeline is 
evaluated against both classes and is measured across three key metrics: 
\textbf{True Positives}, \textbf{False Negatives}, and \textbf{Functional 
False Positives}.


\section{Problem Statement}

Modern systems are increasingly shifting from human-in-the-loop interaction models to machine-in-the-middle execution models. This entails integrating LLM-based agents to programatically invoke tools, APIs, or system resources. Within this setting, model outputs are no longer simply advisory text; they are action-triggers for a live system. 

This transition creates an engineering problem: LLM agents are externally sourced and inherently nondeterministic, yet they are increasingly given authority to initiate privileged operations. Existing integration patterns often conflate agent intent, authorization, and execution. Thus, the introduction of an agent into the control path consequently limits guarantees about policy enforcement, auditing channels, and the level of replay the system supports.

This project addresses the problem through the design and implementation of a security-constrained agent runtime that treats LLMs as untrusted processes. All agent outputs intended to trigger actions are intercepted and evaluated within a deterministic runtime boundary. The runtime strictly enforces explicit security policies. Furthermore, it mediates all tool invocation independently of the agent's internal reasoning or reported intent.

This project does not aim to improve the intelligence of the agent. Rather, the core objective of the project is to provide formal safety, control, and observability guarantees around agent-initiated actions. This is achieved in two ways.

\begin{enumerate}[label=(\arabic*)]
    \item Enforcing explicit, machine-verifiable policies on all proposed actions
    \item Structuring execution in a way that enables systematic verification, auditing, and replay of agent behavior
\end{enumerate}

 \textbf{Engineering Goals}
 \begin{itemize}

    \item Bar agents from direct tool invocation.
    \item Enable systematic auditing and post-hoc verification of agent behavior.
 \end{itemize}
 
  \textbf{Functional Requirements}
 \begin{itemize}

\section{System Design / Implementation}

    The system implements a deterministic mediation approach designed to enforce a hard trust boundary between an untrusted Large Language Model (LLM) agent and the execution of host environment actions.  Unlike traditional frameworks where models directly invoke tool functions, our design inserts a Security-Constrained Runtime as a mandatory intermediary layer. The architecture is founded on a Positive Security Model (Allowlist), where all actions are denied by default unless they strictly adhere to a predefined JSON Proposal Schema. Currently in active development, the system functions as a high-assurance, synchronous orchestrator: it intercepts agent proposals over a local network boundary, validates them, executes approved actions within a contained sandbox, and returns deterministic results or structured errors.\\

   Table II summarizes the capability gaps that motivate the SCR design. In a direct function call pattern, the agent invokes tools without any interception layer, meaning the model is implicitly trusted, enforcement is absent, and no audit trail exists for post-hoc replay. Orchestration frameworks such as ReAct [5] and AgentGuard [4] introduce partial structure through higher-level coordination, but still rely on probabilistic guardrails rather than deterministic constraints, and do not formally separate agent intent from host execution. The proposed SCR addresses all five capabilities by treating the LLM as an untrusted process, enforcing a Positive Security Model through static Zod schemas, and recording every proposal to an immutable audit log, making deterministic enforcement, explicit policy, and replayable execution structural guarantees rather than best-effort properties.\\

\textbf{Design Decisions: }

    To isolate the core research problem of constraining non-deterministic behavior, the system is designed with two strict architectural constraints:

    \begin{itemize}
        \item \textbf{Stateless Execution}: The runtime does not maintain complex session states, database transitions, or multi-client APIs. This stateless design eliminates attack vectors related to session hijacking or state corruption, ensuring each agent proposal is evaluated strictly on its own semantic merits.

        \item \textbf{Static Policy Enforcement}: Rather than using a dynamic policy engine, security policies are statically hardcoded directly into the type-validation layer. Constraints, such as restricting all file operations strictly to a \texttt{/sandbox/} directory and limiting file modifications to \texttt{.txt} and \texttt{.md} extensions, are structurally guaranteed before the execution engine is ever invoked.
    \end{itemize}

\textbf{Design Alternatives:} 

While containerization (e.g., Docker) provides OS-level resource isolation, it does not validate the semantic intent of agent actions. A containerized agent can still issue destructive commands within its sandbox that are difficult to audit. Our approach utilizes a custom Security-Constrained Runtime to enforce \textbf{semantic isolation}. By intercepting proposals at the application layer via Zod-based schema contracts, we convert natural language intent into a rigid, typed API. This ensures that the agent can only interact with the host through a deterministic state machine, preventing the "detonation" of a malicious payload before it ever reaches a system call.

\textbf{Implementation}

    The system implementation integrates a Python environment with a deterministic TypeScript runtime via a local HTTP interface.

    \begin{itemize}
        \item \textbf{Untrusted Agent SDK (Python)}:  The agent layer is implemented in Python. It is responsible for querying the LLM, prompting it to output intents matching our system's JSON schema, and transmitting these intents via HTTP POST requests to the runtime. It supports the \textbf{Gemini Flash and Gemma 3 model family} through system instruction inlining.
        \item \textbf{Runtime-Agent Boundary (Node.js/Express)}: A lightweight local Express server acts as the service ingress point. It catches fatal malformed payloads (e.g., broken JSON) and issues immediate HTTP 400 responses, preventing unhandled exceptions from crashing the runtime.
        \item \textbf{Validation \& Static Policy Layer (Zod}): The TypeScript runtime utilizes Zod for rigorous schema validation before any execution occurs. The schema enforces strict semantic versioning; any mismatch in the MAJOR version (e.g., receiving 2.0.0 when 1.x.x is supported) results in an immediate hard failure and a structured error. To ensure complete auditability, every proposal must globally include a uuid for request traceability and a reasoning string (minimum length of one character) to expose the agent's internal logic.
        \item \textbf{Explicit Action Allow-list}: The runtime employs a strict Positive Security Model, permitting only eight explicitly defined actions: \texttt{THINK}, \texttt{FINISH}, \texttt{READ\_FILE}, \texttt{WRITE\_FILE}, \texttt{DELETE\_FILE}, \texttt{RENAME\_FILE}, \texttt{LIST\_FILES}, and \texttt{CREATE\_DIRECTORY}. Arbitrary system commands (e.g., \texttt{run\_command} or \texttt{spawn\_process}) are intentionally omitted from the core schema, making direct host control structurally impossible.
        \item \textbf{Action Execution Layer (Node.js)}: Validated proposals are routed through a synchronous while loop orchestrator to specific Node.js file system wrappers. These wrappers perform host I/O operations with heavily constrained arguments. For example, all file paths must explicitly begin with \texttt{/sandbox/}. Furthermore, destructive or modifying actions (\texttt{WRITE\_FILE}, \texttt{DELETE\_FILE}, and the destination of \texttt{RENAME\_FILE}) are strictly limited to \texttt{.txt} and \texttt{.md} file extensions to prevent the alteration or deletion of system binaries.
        \item \textbf{Logging and Audit Subsystem}: The architecture incorporates a logging and audit subsystem to ensure complete observability of the agent-runtime interface. The system utilizes a structured log entry schema that captures critical execution metadata, including global proposal identifiers (UUIDs), validation outcomes, authorization statuses, and raw host-execution results. By persistently recording correlated audit events to disk for every step in the execution lifecycle, the runtime provides a verifiable "source of truth" necessary for debugging non-deterministic agent behavior and performing post-hoc security audits. 
    \item \textbf{Execution Error Logic}: The runtime implements a deterministic error-handling lifecycle that maps host-level exceptions to standardized outcomes. Internal failures are categorized as VALIDATION\_ERROR, DENIED, or EXECUTION\_ERROR depending on the failure phase. For execution-stage failures, the runtime calls a handler to determine a containment strategy, such as retrying, prompting for clarification, or aborting execution, based on the specific action and error code encountered. 
    \item \textbf{Evaluation and Metrics Pipeline}: The system features a Python-based evaluation pipeline designed for performance analysis. This module ingests audit traces and utilizes a classification engine to map runtime outcomes against the known intent of the test harness. It computes high-level security metrics, including True Positives (intercepted attacks), True Negatives (successful benign actions), and System Faults, enabling data-driven validation of the security boundary. 
\item \textbf{Proposal Handler}: The runtime employs a multi-stage validation gate to enforce structural and request integrity. The process begins with a "Receive" validation that enforces a 1024-byte payload limit and rejects any proposals containing null bytes or non-printable ASCII characters. A dedicated handler then performs a stateful check for ID collisions by cross-referencing incoming request identifiers against historical audit logs, ensuring that each agent intent is unique and preventing replay attacks. 
    \end{itemize}

\section{Evaluation and Results}

\subsection{Testing Strategy}

We evaluate our system as a binary decision system that classifies each agent proposal as either \textit{allow} or \textit{deny}. For each test case, we define an expected decision based on our policy rules and compare it with the actual output of the system. These testing categories do not represent separate test cases, but rather different aspects of system behavior that we aim to verify. A single test case may be used to evaluate multiple aspects at the same time.

\textbf{Unit Testing.}  
We verify that security policies are correctly enforced at the host-action execution layer. 
This includes validating absolute path resolution logic to prevent directory traversal 
and ensuring that file system wrappers strictly enforce sandbox boundaries and file extension constraints (e.g., \verb|.txt| and \verb|.md| allowlists). These tests ensure that the execution engine behaves correctly under well-formed inputs 
and prevents out-of-bounds I/O.

\textbf{Validation Testing.}  
We verify the structural integrity of incoming agent proposals prior to execution. This includes validating JSON format correctness, required field presence, type constraints, and schema version compatibility. 
These tests ensure that malformed or non-compliant inputs are rejected at the ingress boundary before reaching the execution layer.

\textbf{Adversarial Testing.}  
To simulate a compromised  LLM, we implemented a suite of \textbf{110 specific fault-injection cases}. These tests subject the execution boundary to adversarial payloads and exploits, including path traversal attempts and arbitrary code execution via file overwrites. 

\textbf{Determinism and Reliability Testing.}  
We verify system determinism by executing identical proposals across multiple environmental states. This phase was critical for identifying and neutralizing a non-deterministic race condition in the server initialization phase, ensuring the runtime properly awaits sandbox provisioning before accepting ingress traffic.

\textbf{Audit Logging.}  
The testing harness utilizes a decoupled Python-based evaluation pipeline to ingest immutable audit logs. This allows for the classification of every interaction as a True Positive, True Negative, False Positive, or System Fault, providing a reproducible summary of system performance.


\subsection{Evaluation Metrics}

We treat the runtime as a binary classification problem, where each input is either allowed or denied.

We use the following metrics:

\begin{itemize}

    \item \textbf{True Positive (TP):} An adversarial or malformed proposal correctly intercepted.
    \item \textbf{True Negative (TN):} A valid, safe action correctly executed.
    \item \textbf{False Positive (FP):} A benign request incorrectly denied due to strict schema enforcement.
    \item \textbf{False Negative (FN):} A malicious request incorrectly allowed (security failure).
    \item \textbf{System Fault (SF):} An infrastructure error occurring during processing. 
\end{itemize}


\textbf{Automated Evaluation Pipeline}

We developed a Python-based evaluation pipeline to transform raw logs into structured performance data.

\textbf{Log Classification Engine} The pipeline ingests audit logs generated by the Node.js runtime. A classification module maps the outcome of each step (Success, Denied, Validation\_Error, or Execution\_Error) against the known intent of the test harness (Benign vs. Malicious). This allows for the automated computation of True Positive Rate, False Positive Rate, and System Fault rates across high-volume test suites.

\textbf{CI/CD Integration and Observability} The evaluation framework is fully integrated into the GitHub Actions CI/CD pipeline. For every system change, the environment executes a 117-step stress test and generates a Markdown-formatted performance report. This ensures that any regression in security (False Negatives) or stability (System Faults) is identified immediately. By aggregating results across multiple test runs, we established a  dataset of 364 interactions to validate the system's final performance.

\subsection{Results}

To validate the architecture, the evaluation and metrics pipeline was executed against a multi-agent simulation environment, yielding an aggregate dataset of \textbf{364 agent interactions}. This volume allows for a  assessment of the runtime's security boundary and infrastructure stability across the development lifecycle. 

\textbf{Infrastructure Stabilization} Initial evaluation (V2 and V3) revealed significant non-determinism in the system's reliability, with infrastructure-level fault rates fluctuating between 14.2\% and 42.8\%. Log analysis and execution tracing identified a race condition in the server initialization phase: the asynchronous sandbox bootstrapping function was not being properly awaited before the runtime listener began accepting ingress traffic. Depending on the test-environment speed, the harness would occasionally issue file-system requests before the sandbox directory was provisioned by the host OS, resulting in unhandled ENOENT (Path Not Found) exceptions. 

To neutralize this jitter, the Action Execution Layer was refactored to enforce a synchronous bootstrap loop and transition to absolute path resolution. Following these architectural refinements, the aggregate System Fault Rate stabilized at a deterministic \textbf{9.4\%}. Telemetry confirms these remaining faults are primarily host-state sensitivities—such as a \verb|READ_FILE| operation targeting a non-existent file—rather than failures of the mediation layer itself. 

\textbf{Adversarial Stress Testing (Jailbreak Mitigation)} A core component of the evaluation was the integration of a dedicated adversarial driver, \verb|run_llm_fault_test|. This module utilizes a non-deterministic LLM agent to generate 110 dynamically varied adversarial proposals across 11 fault categories. This approach simulates real-world exploits, such as prompt injection intended to achieve arbitrary code execution or complex path traversal to escape the sandbox.

The runtime maintained a \textbf{0.0 False Negative Rate (FNR)} throughout these tests, correctly identifying and intercepting 100\% of the 136 malicious payloads encountered in the aggregate dataset. By enforcing strict Zod-based schema contracts at the mediation boundary, the system ensured that no adversarial intent resulted in a host-level security breach.

\textbf{Functional Success and Utility Trade-offs} The runtime successfully processed its first benign payloads in the final phase, demonstrating that the system can distinguish between deterministic execution and malicious exploitation. While the aggregate False Positive Rate (FPR) remains high at \textbf{94.3\%} on benign intent, log analysis confirms these are "parsing successes" rather than policy failures. The runtime correctly rejected these proposals because the probabilistic LLM output failed to adhere to the rigid JSON contract (e.g., missing mandatory \verb|action| or \verb|reasoning| fields). This confirms that while the Positive Security Model provides an impenetrable security boundary, it imposes a high requirement for structural precision on the agent layer.

\subsection{False Positive Analysis}

Across the aggregate evaluation dataset (N=364), the runtime exhibited a  False Positive Rate (FPR) of 94.3\% on benign intent. While initial manual validation in controlled environments yielded a lower 0.33 FPR, high-volume stress testing using the Gemma 3 model family revealed a fundamental \textbf{semantic-structural mismatch} between probabilistic LLM outputs and our deterministic mediation layer.

\textbf{Parsing Successes vs. Policy Failures} Post-hoc log analysis confirms that these False Positives are categorized as "parsing successes" rather than failures of the security logic. Because the Security-Constrained Runtime (SCR) utilizes a \textbf{Positive Security Model}, it treats any structural deviation as a potential exploit. The majority of rejections were triggered because the LLM failed to adhere to the strict Zod-based JSON contract—often omitting mandatory fields such as \verb|action| or \verb|reasoning|. In a high-assurance security environment, the runtime correctly prioritizes system integrity over functional flexibility by denying these malformed proposals.
 
\subsubsection{Root-Cause Breakdown by Action Type}

To identify the specific drivers of utility loss, we analyzed the distribution of rejections across the action registry using a targeted 117-entry stress-test suite.

\textbf{Complex Actions (WRITE\_FILE, LIST\_FILES)} These actions generated the highest volume of False Positives. Rejections were primarily driven by the strict schema requirements for path arguments and content fields. The LLM frequently produced "hallucinated" path formats or omitted mandatory keys, which the Zod validator correctly intercepted as schema violations to prevent potential out-of-bounds execution.

\begin{itemize}
    \item \textbf{LIST\_FILES:} Rejections were frequently caused by missing or improperly formatted path arguments.
    \item \textbf{WRITE\_FILE:} Failures often stemmed from missing content fields or unauthorized file extensions.
\end{itemize}

\textbf{Structural Omissions} A significant portion of rejections occurred when the LLM failed to output a valid action identifier (e.g., returning "N/A" or a markdown-wrapped code block). The runtime's ingress handler correctly identified these as structural failures, preventing malformed or ambiguous data from ever reaching the execution orchestrator.

\textbf{Lightweight Actions (THINK, FINISH)} Even simple internal actions contributed to the FPR due to the mandatory "reasoning" constraint. Proposals that omitted the internal logic string were denied, reinforcing the system's requirement for a complete and auditable execution trace.

\textbf{Engineering Conclusion} The high FPR is a direct artifact of the runtime's uncompromising security posture. These results demonstrate that while the SCR provides an impenetrable boundary against adversarial exploits (0.0 FNR), its operational utility is gated by the LLM's ability to maintain structural precision. This identifies a critical design trade-off: absolute safety at the mediation boundary currently requires a level of formatting rigor that exceeds the reliability of current probabilistic models.

\section{Limitations}

\textbf{1. Structural Enforcement vs. Semantic Isolation}

A primary limitation of the current evaluation centers on the runtime’s inherent prioritization of structural validity over semantic assessment. While the system achieved a 0.0\% False Negative Rate (FNR) across all adversarial tests, the majority of malicious payloads were intercepted by the Zod validation gate due to formatting errors before semantic evaluation could occur. Although this indicates that the results do not cleanly isolate the semantic analyzer’s performance against properly formatted malicious payloads, this behavior is an intentional success of the Positive Security Model. In a high-assurance runtime, intercepting malformed adversarial intent via structural enforcement is a core design goal. This demonstrates the system’s ability to neutralize the payload at the earliest possible ingress point, even if the rejections are categorized as validation errors rather than semantic denials.

\textbf{2. The Security-Utility Bottleneck (False Positive Crisis)}

The reported 94.3\% False Positive Rate (FPR) represents a critical trade-off between security rigor and functional utility. While this metric validates the strictness of our Positive Security Model, it simultaneously identifies a significant operational friction point for production environments. A system that blocks nearly all benign requests due to minor formatting non-determinism represents a robust enforcement mechanism but requires deliberate calibration to achieve operational utility. This high friction suggests that while the enforcement logic is structurally sound, the current implementation lacks the smoothing required to bridge the Structural-Semantic Gap without introducing prohibitive operational overhead.

\textbf{3. Statelessness and the "Entropy Sink" in Chained Tasks}

The current prototype utilizes a strictly stateless execution model. While statelessness is an intentional security feature (preventing state-injection or persistence-based escapes) it introduces a significant limitation for complex, multi-step engineering tasks. In a "machine-in-the-middle" workflow, a single structural error at any step in the sequence triggers a hard failure. This identifies an entropy risk, as the number of steps in a chain increases, the cumulative probability of a schema violation approaches 1.0, making long-running autonomous tasks statistically unlikely to complete without frequent human intervention.

\textbf{4. Empirical Scope and Hypothesized Generalization}

While the \$N=117\$ evaluation dataset provided a sufficient volume to identify performance trends and stabilize the system fault rate, it does not constitute a statistically exhaustive audit. The results are limited to the specific action allowlist and filesystem constraints defined in the current Zod schema. Furthermore, the evaluation was conducted using a specific family of models (Gemini and Gemma); the system's performance, particularly the False Positive Rate, is likely to fluctuate significantly depending on the underlying model's capability for constrained decoding and structural adherence to the defined contract.

\textbf{5. Defensive Gap: Indirect Prompt Injection}

The SCR is designed to secure the host environment from the agent, but it does not inherently protect the agent from the environment. The system currently lacks a mechanism to sanitize the inputs the agent receives from the host (e.g., reading a malicious file that contains an indirect prompt injection). While the runtime would likely block the \textit{resulting} malicious action, the agent’s internal reasoning could still be compromised, leading to a loss of functional integrity that is not currently captured by our structural metrics.

\section{Conclusion}

In this project, we designed and implemented a Security-Constrained Runtime (SCR) to serve as a deterministic safety layer between autonomous LLM agents and the host system. The architecture is predicated on a hard trust boundary that logically decouples agentic reasoning from system execution. By enforcing a Positive Security Model (PSM) through strict Zod schemas, static action allowlists, and sandboxed filesystem constraints, the runtime ensures that malformed requests, unauthorized system calls, and unsafe path operations are structurally intercepted before they can reach the execution environment.

The efficacy of this mediation layer is supported by a diagnostic framework that provides the forensic observability necessary to iterate on non-deterministic system behavior. A critical phase of this engineering process involved neutralizing a systemic race condition in the server initialization layer, where asynchronous sandbox provisioning caused the System Fault Rate to spike to 42.8\%, by refactoring the Action Execution Layer to enforce a synchronous bootstrap loop and absolute path resolution. This stabilized the fault rate at a final 9.4\% across a longitudinal dataset of 364 aggregated test cases, including 110 adversarial stress tests.

Performance validation yielded a 0.0\% False Negative Rate, confirming that no adversarial intent bypassed the trust boundary. While these interceptions were frequently driven by the structural rigor of the Zod schemas rather than semantic intent analysis, the host environment remained protected throughout all testing cycles. However, it is critical to acknowledge the fundamental "Structural-Semantic Gap" revealed by the 94.3\% False Positive Rate. While these blocked actions validate the system’s security rigor, they also represent a significant operational bottleneck. A system that denies the majority of benign requests demonstrates a robust enforcement mechanism but has not yet reached the threshold for production utility.

Future work must bridge this gap through policy calibration to ensure that deterministic safety does not come at the absolute cost of functional throughput. Ultimately, the SCR proves that a deterministic mediation layer can maintain total security enforcement even against adversarial models. By converting natural language intent into a rigid, typed, and auditable API, the system provides the non-negotiable safety and observability guarantees required for machine-in-the-middle workflows in mission-critical and deep-tech sectors.

\section{Project Postmortem}

\subsection{Process \& Outcomes Reflection}

\subsubsection{\textbf{Proposed vs. Actual Process}}

The initial engineering roadmap was designed for a four-person team with a focus on breadth, encompassing a multi-agent simulation harness, persistent state models, and a complex client-facing API. Following the departure of a key team member, the project transitioned was descoped from a feature-rich model to a Minimum Viable Product. Instead of building multiple agents with varied tool access, we pivoted to harden the core trust boundary between a single agent and the runtime.

This process shift was driven by the realization that securing a non-deterministic agentic workflow requires significantly more architectural rigor than originally anticipated. We moved away from probabilistic guardrails, which were the initial proposed outcome, and toward a Positive Security Model (PSM). This  change in process meant that our engineering efforts were redirected from orchestration logic toward the structural enforcement of deterministic schemas.

\subsubsection{\textbf{Proposed vs. Actual Outcomes}}

The actual outcome of the project is an architecture that prioritizes system integrity over functional throughput. While the proposed outcome included a wider array of integrated tools and MCP connections, the actual prototype provides a much more rigorous verification surface. Specifically, the development of the adversarial stress-test suite far exceeded the original evaluation plan. The discovery of the "Structural-Semantic Gap" was an emergent outcome. We found that the efficiency of a deterministic runtime is bottlenecked not by the security logic itself, but by the model's ability to maintain formatting under adversarial pressure.

\subsection{Team Retrospective}

\subsubsection{\textbf{What went well}
}

\begin{itemize}
   

 \item One of our largest accomplishments was completing a baseline implementation of our SCR by the V1 deadline. Having a functional prototype early allowed us to spend the remainder of the project polishing feature implementation and conducting robust stress testing to ensure our model was as structurally sound as possible.

 \item In addition to completing a baseline implementation ahead of schedule, integrating automated test infrastructure and LLM-powered agentic code reviews for GitHub Pull Requests marked a significant improvement in our testing and debugging workflow. During V1, we spent considerable time integrating final changes and verifying the system behaved as intended. While this approach had its merits, upgrading to an automated testing flow made the process far more efficient and enabled our team to maintain a true CI/CD methodology throughout development. 

\end{itemize}
\subsubsection{\textbf{What went poorly}
}
\begin{itemize}
    \item The initial task estimation underestimated the amount of debugging and integration effort required late in development. This would have been even more apparent with the original feature-rich proposed project.
    \item The engineering process focused so heavily on enforcement that we failed to allocate sufficient time for semantic calibration. This resulted in a high False Positive Rate. While this is a security success, it represents a process failure in balancing functional utility with structural rigor. Earlier integration testing with a more diverse range of agent personas might have identified this bottleneck sooner.
\end{itemize}
